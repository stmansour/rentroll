Setting Up A New Business With CSV Import

October 26, 2016

Version 0.1

With Accord RentRoll you can create all the information about your business using the web browser interface.  But this can be a time-consuming process if there are a lot of rental items to enter. An alternative is to use Comma-Separated-Value (CSV) files to preload this information.  CSV files can be easily created from popular spreadsheet programs such as Microsoft Excel.  You can use CSV files to load some or all of your company information.  This guide will walk through how to import your data to Accord Rentroll. You can pick and choose the CSV files you want to load and use the interface for everything else.

The order in which information is loaded is significant. Certain information must already be in place to load other information.  For example, information about a business must be loaded before rentables belonging to that business can be loaded. Fortunately, the interface for the CSV Loaders is ordered from top to bottom in an order that works well. The order in which information is presented in the sections below follows the interface and should work fine for most businesses.

Input Data Format Conventions

Unique Identifiers, or IDs, are shown as the letters and an 8-digit 0-filled number.  For example, RT00000001 is the ID for a Rentable Type with the identifier 1. The number of leading zeros is arbitrary. That is RT00000001, RT001, and RT1 all refer to the Rentable Type with id 1.

Date values in the CSV files can be specified in any of the following formats:

01/02/06

1/2/06

1/2/2006

2006-01-02

2006-01-02 15:04:00 MST  (for date-time values)

Input values of “yes” and “no” can be entered in several ways.  All yes/no values are considered case-insensitive.  This means that values such as “yEs”, “YEs”, and “yes” are all the same.  Yes values can be:  yes, y, or 1.   No values can be:  no, n, or 0.

Basic Plan Of Attack

As of this writing, the people using Rentroll will probably be using the Functional Testing Interface. The command line instructions are listed at the bottom of each section. If you’re using the web browser interface you can ignore this inforamtion. The web browser interface available today is not the product’s user interface. It is more of a functional testing interface. It is not a very intuitive interface. But it does make many basic commands available. In fact, there are administrative commands available that will not be available in the final product. While these commands are very useful, you can easily shoot yourself in the foot with them. So use them with caution as directed.

A good approach to using this interface is as follows:

  1. Make a backup of the database before you start.
  2. Use the CSV import commands to import data and the report commands to verify that the data is correct.
  3. If something goes wrong -- the data gets messed up in some way -- you can always restore the system to a known-good working state by restoring your backup
  4. If you import data and check everything out thoroughly and decide that everything looks good, then make another backup -- they are very small and efficient, it’s better to have backups that we don’t need than not to have them when you need them.

The following sections discuss how to use a few of the key commands.

The CSVLoad Interface

For loading any of the supported CSV files, the CSVLoad interface can be used. Click the CSVLoad button on the main screen in the “CSV Load” column. You will see a screen that looks something like the following (though it may change as development is ongoing):

Each row contains three columns of controls.  The first column describes what kind of a CSV file you can load. The second column is the file selector. Click the Choose File button to get a system dialog for opening a file then select the .CSV file you want.  After you’ve selected the file and clicked the “Open” button the file name will appear just to the right of the Choose File button.  Next, click the button in the third column to load the CSV file into the database.

Side Note: These buttons are labelled with some letters. These letters are typically the prefix to the unique identifiers associated with the data type you are loading.  For example, the unique identifier (ID) for a Business in Rentroll is presented as the character ‘B’ followed by a zero-filled number. The first business created in Rentroll will have the ID B00000001. So ‘B’ is the prefix for Business IDs.  Similarly, Rentable Types have IDs of the form ‘RT00000085’, its prefix is ‘RT’.  And so on.

Sometimes, more information than the filename is needed. Once the data is loaded information, including any error messages, is displayed in the area below the buttons. You may need to scroll your browser down to see it.

Assessment Instances

While the CSV Load capability was designed to speed up the process of setting up a new business in Rentroll, it was found to be a great boon to automated testing. Thus, there are some CSV Loaders for data types that may not be expected. For example, Assessments and Receipts can be loaded via the CSV Load capability.  It is important to understand how to work with Assessments as we test and production-harden Rentroll. In particular, recurring Assessments require some explanation and special handling.

Assessments can recur daily, weekly, monthly, and at many other intervals. When you define an Assessment, you give it an attribute that defines how it recurs.  The recurring assessment is saved and as its recurrences happen, new instances of the recurring assessment are created. Payments are applied to the instances.  Journal and ledger entries refer to assessment instances.  Reports typically show assessment instances.  In the final product, these instances will be generated automatically by the RentRoll server as soon as the time for an instance arrives.  However, for testing purposes, we may not want to wait for days or weeks or months for an instance time to arrive so that we can test it.  

To deal with this problem, the functional test interface provides date controls that can be set to any date or date range desired. Using the command “Update Journals” forces the creation of any recurring assessment instances in the date range defined by the current settings in the date controls. If the recurring instance was previously created, it will not be created again -- thus, it is idempotent and your results will not change no matter how many times you press the “Update Journals” button.  This kind of functionality will likely not be available on the final product.

Vacancy

Vacancy detection is done automatically by the Rentroll server. At the time of this writing, we plan to have vacancy detection run at the end of each Rentable’s rent cycle. So, Vacancy journal entries will only appear at the end of a rent cycle.

However, at the time of this writing, Vacancy detection is not done automatically in this functional test version of RentRoll. It is performed on-demand by pressing the Update Vacancy button on the main interface. This enables you to test the vacancy capability without having to wait until the end of the rent cycle.

String Lists

A String List is a named collection of text strings. Each string in a string list can be uniquely identified. The strings can be used for questionnaires and other selection choices that may occur in your business.

As part of collecting information on your customers there will be questions that you ask them that can have a wide variety of answers. And you will not always want individual, unique answers. You may wish to categorize the answers so that you can easily process the information into useful statistics.  Examples include questions like, “how did you hear about us” or “what made you decide to use our services”, etc. To enable you to bucketize answers to questions like these, RentRoll enables you to create lists of text strings, where each text string belongs to a group and can be uniquely identified.  

As an example, suppose you want to find out during the application process how a customer found out about your apartment complex. You can create a question about this on your application form and provide a list of choices from which the customers can choose an answer. For this purpose, we could define a String List called HowFound and add to it a number of choices about how a customer may have heard about them. This question along with the String List of answers could then be added to the application form a customer fills out. You can get a report showing the total number of customers that chose each answer.

Some example String Lists have been created in the file strlists.csv. You can copy this file and modify it to suit your company.

The fields of a String List are defined as follows.

Attribute

Description

BUD

Business Unit with which this String List is associated

Name

The name of the String List. Following the example above, we can name a string list HowFound and fill it with a number of different choices about how a customer found them.

Value

Any sequence of characters.  For example, if you wanted to know how many people found out about your apartments through apartmenthomeliving.com you might add the string “Apartment Home Living.com” to the HowFound stringlist.

Note that sometimes you may have so many choices for a particular StringList that a single sequential list of choices might be overwhelming to the user.  In this case, you may wish to categorize choices in a way that makes them easier to find.  RentRoll’s user interface will use the categories to make the user interface easier for customers to choose.  For example, an apartment complex might advertise in the newspaper, in magazines, on television, via direct mail, and on the internet.  You can specify a category for each string using the caret (^), for example:

“Internet ^ Apartment Home Living.com”

This would cause the string “Apartment Home Living.com” to be categorized under the “Internet” category.

Make a copy of the sample strlists.csv file and update it for your business.

RUserSpec

RUserSpec defines the users who are associated with the rentable during a particular duration of time. The general form of the specification is:

user1,start-date1,stop-date1;user2,start-date2,stop-date2...

The basic pattern is a 3-tuple consisting of a user, a start date, and a stop date separated by a commas (,). Multiple 3-tuples can be expressed by separating them with the semi-colon (;).

Users can be entered with any of several unique identifiers that can be found in their Transactant information:

  • TransactantID --  TC0003422 (or similar)
  • Phone number -- for example, their cell phone number
  • Email address

As an example, suppose that Bill (bill@example.com) is using a rentable from Jul 7, 2016 through August 1, 2017. The RUserSpec for this would be:

bill@example.com,7/7/16,8/1/17

Multiple Payors can be added to the PayorSpec by using a semi-colon (;) to separate them.  For example, suppose that Sue was also going to use the rentable during that time period. The RUserSpec would be:

bill@example.com,7/7/16,8/1/17;sue@example.com,7/7/16,8/1/17

 

Note that users can also be assigned to rentables in the Rental Agreement CSV file (see below).  

RentableStatus

A rentable status is defined by a status type, a start date, and a stop date.  The status values are as follows:

  • 0 = unknown
  • 1 = online, operational
  • 2 = admin,
  • 3 = employee
  • 4 = owner occupied
  • 5 = offline, not operational

A Rentable can have multiple status values over time. The general format for entry is:

“status1,startDate1,stopDate1;status2,startDate2,stopDate2;...”

In most cases, a Rentable will have only one state, and its value will usually be “ONLINE”.  Such an entry would look like this:

“1,2015-02-01,2017-02-01”

This would mean that the rentable went online starting February 1, 2015 and that it will stay online until February 1, 2017.  If you would like to leave the end date unspecified, you can omit the last date -- but do not omit the comma. Here is an example:

“1,2015-02-01,”

This can be interpreted as the Rentable being online and it stays online until it is updated and put it in a different State.

RentableTypeRef

RentableTypeRef defines what type the rentable is for a period of time. As an example, you may rent hotel rooms -- sometimes the room may be rented on a daily basis for transient occupancy, at other times it may be rented on a monthly basis for longer term occupancy. You can change the rentable type as often as needed.

The general form for RentableTypeRef in the CSV file is:

“RTname,dtStart1,dtStop1;RTname2,dtStart2,dtStop2;...”

Where:

  • RTname is the RentableType name
  • DtStart is the start date/time for this rentable (typically the start date of the rental contract)
  • DtStop is the stop date/time for this rentable (typically the stop date of the Rental Agreement)

As an example, suppose there are two RentableTypes.  One is named A2 and it is used for hotel rooms rented by the day.  The other is named B2 and is used for hotel rooms rented by the month. If you have a room that was rented by the day for the first 10 days of the month, then it changed to being rented by the month it could be entered as follows:

“A2,1/10/16,6/1/16;B2,6/1/16,6/1/18”

This says that the rentable will be of RentableType A2 from Jan 1, 2016 up to but not including Jun 1, 2016. On Jun 1, 2016 its type is B2 and stays that way until 6/1/18. Sometimes the end date is not known. By omitting the stop date on the last RentableTypeRef, the rentable will remain that RentableType until someone updates it and sets it to another type. That entry looks like this. Note that the final comma is required:

“A2,1/10/16,6/1/16;B2,6/1/16,”

  • Their email address

The generic form for the Payor spec is:

payor1,start-date1,stop-date1;payor2,start-date2,stop-date2...

As an example, suppose that Bill (bill@example.com) is a both renter and the Payor for the his apartment rent over the duration of the Rental Agreement, which is Jul 7, 2016 through August 1, 2017. The PayorSpec for this would be:

bill@example.com,7/7/2016,8/1/2017

If the Rental Agreement Start / Stop time were also July 7, 2016 and August 1, 2017 then the date fields could be blank as they default to the Rental Agreement Start / Stop dates. Thus, the PayorSpec would simplify to:

bill@example.com,,

Multiple Payors can be added to the PayorSpec by using a semi-colon (;) to separate them.  For example, suppose that Bill got had a room-mate, Bob (123-456-7890, bob@acme.com)  and both of them were payors. This could be entered as follows.

bill@example.com,,;123-456-7890,,

or

bill@example.com,,;bob@acme.com,,

This assumes that the date range is the same as the Rental Agreement Start/Stop time. The PayorSpec below is identical in effect to the ones above:

bill@example.com,7/7/16,8/1/17;bob@acme.com,7/7/16,8/1/17

Remember that the dates can be fully specified if necessary. If Bill was the Payor for 2016 and Bob was the Payor for 2017 the PayorSpec would look like this:

bill@example.com,7/7/16,12/31/17;bob@acme.com,1/1/17,8/1/17

UserSpec

The UserSpec identifies the users of the Rentables under this Rental Agreement. The format is identical to the PayorSpec described above:

user1,start-date1,stop-date1;user2,start-date2,stop-date2…

As an example, suppose that Bill (bill@example.com, cell: 123-456-7890), his wife Sue (sue@example.com, cell: 123-456-7891), and their daughter Mary are the users of a rental house.  Mary does not have a phone number or email address, but she can still be identified by her Transactant ID (TC00003562). The UserSpec for this family would be:

bill@example.com,,;sue@example.com,,;TC3562,,

This assumes that the dates are all identical to the start/stop of the Rental Agreement, which is the most common scenario. But the start and stop date of any user can be specified. As an example, suppose that Mary was coming to live with Bill and Sue beginning September 1, 2016. The UserSpec would be:

bill@example.com,,;sue@example.com,,;TC3562,9/1/16,

NOTE: Users identified here are applied to all rentables that the Covered by the Rental Agreement; those called out in RentSpec below.  See notes in RentSpec for options.

Renewal

1 - rent cycle period automatic renewal

2 - use lease extension options prior to the Agreement Stop Date

SpecialProvision

Enter any special provisions to associate with this Rental Agreement. This field can be up to 1024 characters in length.

RentableSpec

The RentableSpec defines the Rentables and their associated Contract Rent covered by the Rental Agreement. The generic form of the RentableSpec is

rentable1,contractrent1;rentable2,contractrent2...

This spec consists of pairs of data. Each pair consists of a Rentable, identified by its Name, and its associated Contract Rent. These two values are separated by a comma (,).  If more than one pair is needed, separate each pair by a semi-colon (;).  Here is an example:

U101,2500.00;U102,2350.00

This specifies two rentables, U101 which rents for 2500.00 and  U102 which rents for 2350.00.

NOTES:

Each person in the UserSpec is added automatically as a user of each Rentable called out in the RentableSpec. There may be times when certain users should not be associated with particular Rentables. There are two options for dealing with this. Option 1 is to leave the UserSpec blank in this CSV and call out the users in the Rentable CSV file.  This will allow you to assign users Rentable by Rentable.

Use care when the Rentable names are numbers and Excel.  Consider this example:

101,1350.00        ERROR! Excel saves as 1,011,350.00

This RentableSpec defines a single Rentable named “101” with a Contract Rent of $1350.00 every rent cycle. If you enter this information into Excel, it will make the assumption that you’ve entered the comma in the wrong place and it will change the entry to the numeric value 1,011,350.00. To ensure that your entry is interpreted as a string, put a space after the comma. This signals to Excel that it is a string:

101, 1350.00       This works!

Notes

Adds the supplied note to the Rental Agreement

Invoices

An invoice is one or more Assessments associated with a particular Rental Agreement that have been grouped and issued to a payor indicating the amount owed.  Once an Assessment has been added to an Invoice it cannot be modified. Recall that recurring Assessments have instances which are used for billing purposes. Only a non-recurring instances of Assessments can be included in an Invoice. The attributes of an Invoice are shown below.

Attribute

Description

BUD

Business Unit

Date

Date the invoice was created

PayorSpec

This defines the Payors to whom the Invoice will be sent. If this field is blank, the invoice will be sent to all the Payors associated with the Rental Agreement to which all the Assessments of the Invoice belong. If there are any other payors for this Invoice they can be specified as a comma separated list of identifiers. The identifiers can be the Transactant ID (TCID), the person’s email address, or the person’s telephone number.  The general format is:

payor1,payor2, …

Here is an example:

TC0003,4,bill@example.com,512-123-4567

This example indicates that the person whose Transactant ID is 3, the person whose Transactant ID is 4, the person with email address bill@example.com, and the person whose phone number is 512-123-4567 along with all the Payors for the Rental Agreement will receive a copy of this Invoice.

DateDue

The date on which the payment for the amount of the Invoice is due.

DeliveredBy

A string indicating how the Invoice will be delivered.  Examples include: US Mail, FedEx, UPS, ...

AssessmentSpec

A comma separated list of Assessment IDs that make up the Invoice. All Assessments must refer to the same Rental Agreement. The general form is:

assessment1,assessment2,...

When specifying an Assessment ID, you can use “ASM” followed by  an ID number with optional leading 0s. Or just list the number by itself. Thus,  ASM000001, ASM01, ASM1, and 1 all refer to the Assessment with ASMID 1.  An example:

ASM4,5,ASM7,ASM00010

specifies Assessments with IDs 4, 5, 7, and 10.
